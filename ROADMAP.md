# frame — Roadmap

A Wayland-native screenshot and annotation tool for MangoWM. Built for a single user. Replaces `grim` + `slurp` + `satty` with a unified, scriptable Rust binary that also closes the scrolling-capture gap on Wayland.

## Project Principles

- **BYOS (Build Your Own Stuff).** Audience of one. No configurability surface beyond what the author uses. No support for use cases the author hasn't asked for.
  - **"Audience of one" is not "no readers."** There are four: the author now, the author later, the Claude session working on this now, and the one working on it later. Documentation that serves those four — decisions, corrections, and the reasoning behind them — is in scope and earns its keep. What's out of scope is documentation for users and contributors who don't exist: getting-started guides, contributor docs, API surface for consumers, support material.
- **Single self-contained binary.** One executable — no plugin directories, no interpreter, no installed data files, no companion processes to deploy. It is **not** statically linked, and never needed to be: the graphics, Wayland, and input stacks are `dlopen`ed at runtime rather than linked, so they never appear in `ldd`. The constraint that actually matters is that `frame` requires no system library Void does not already provide by default. GTK is eliminated by this requirement. Exact linkage is recorded under the `eframe` entry in Architecture Decisions.
- **Clarity over cleverness.** Minimal, precise implementations preferred over flexible, abstract ones.
- **Wayland-first.** No X11 compatibility shim, no legacy fallbacks.

## Architecture Decisions (Locked)

- **Implementation language:** Rust.
- **Output format:** PNG, with optional sidecar JSON for editable annotations.
- **Interactive-mode coordination:** Unix socket at `$XDG_RUNTIME_DIR/frame.sock`. A second invocation of `frame` while a session is active signals the running instance (e.g., scroll-capture stop). The pattern is reusable for any future modal capture; designed as a small command bus rather than a one-off.
- **Annotation editor model:** ordered operation list with a pointer.
  - Undo: move pointer back, re-render from base image through ops up to pointer.
  - Redo: move pointer forward.
  - New op after undo: truncate past pointer, append (standard editor behavior).
  - Save: serialize the op list.
  - Reopen: load list, pointer at end — cross-session undo for free.
  - Each operation must be self-contained: type plus parameters sufficient to re-execute against the base image. No mutable references to prior state.
  - Destructive ops (crop, blur, pixelate) live in the same list as vector ops with different op types. Crop adjusts the canvas rect for subsequent ops; the renderer handles that consistently.
  - Re-render on every change. No diff-based optimization. Undo depth uncapped.
- **Sidecar JSON format:** human-readable. For BYOS debugging, readability beats file size.

## Tier 1 — v1 Ship Target

### Capture
- Region capture, with crosshair and magnifier during selection — **shipped 2026-07-16** (`bf0c4e4`, `216684c`). `frame region`: frozen-grab backdrop, dim, `+` cursor marker, drag rectangle, magnifier loupe; crops to PNG + clipboard. Esc and sub-8px selections cancel with no output.
- Window capture (single-window selection) — **implemented 2026-07-20, verified in use.** `frame window` grabs the active output, raises the region overlay in a window-picking mode over the frozen grab, and crops to the chosen window before handing it to the Quick Access Overlay. Window rectangles come from `mmsg get all-clients` (`window.rs`), mapped from global-logical coordinates into physical grab pixels.
  - **The overlay is one surface with two modes, not two overlays.** A private `Mode` enum (`Region` | `Window(Vec<Rect>)`) in `overlay.rs`; `select_region` and `select_window` are wrappers over a shared `run()`. They share the layer surface, frozen backdrop, frame pacing and exit paths, and differ in exactly three places: which rectangle is highlighted, whether a press sets a drag anchor, and what a release commits. Crosshair and magnifier are region-only — they are aiming aids, and a picker offering fixed rectangles has no pixel to aim at. The layer-shell namespace differs per mode (`frame-region` / `frame-window`) so compositor rules can tell them apart.
  - **Occlusion is captured, and cannot not be.** `zwlr_screencopy_v1` copies an *output*, not a surface, so the crop carries whatever overlapped the target window. Capturing the window's own buffer would need a protocol wlroots does not offer.
  - **Partly-scrolled windows are clamped to the visible part — also forced, not chosen.** The off-viewport remainder was never composited onto an output and exists only in the client's surface buffer, which is unreachable. The picker does not lie about this: the highlight is the clamped rectangle, so what lights up is what you get. The alternative is refusing to offer such windows at all, which loses a capture that currently works.
  - **Overlap resolves floating-above-tiled, then smallest-area.** The hit-test takes the first rectangle containing the cursor, so ordering *is* the stacking policy. The first attempt ranked on area alone, on the intuition that the innermost window wins; testing on 2026-07-20 disproved it. Under the scroller layout tiled windows are columns and a floating window usually spans several of them, making it the *largest* rectangle in an overlap — so floating windows were unpickable wherever they covered tiled ones. mmsg reports no z-order, but it does report `is_floating`, and MangoWM draws floating above tiled; that one real fact now leads, with area surviving as the within-rank tiebreak where innermost-wins does hold.
  - **The crop is the geometry rectangle exactly.** The MangoWM border is compositor chrome, not content.
  - **Empty candidate lists exit rather than raising a picker that cannot be satisfied** — no focused output, or no window passing the filters (same monitor, not minimized, tag intersecting the monitor's `active_tags`, rectangle intersecting the viewport, at least `MIN_SELECTION` per side), each print a message and exit non-zero.
  - **Verified in use 2026-07-20:** tiled windows, a partly-off-viewport window (clamped as designed), and a floating window over tiled ones after the ordering fix. **Unverified:** the cancel paths, which are shared with the shipped region mode and differ only in that a release over bare desktop yields `None` like a degenerate drag. Dual-display inherits the same open question as `full`.
- Fullscreen capture — **implemented 2026-07-17, single-display verified; dual-display verification pending hardware.** `frame full` grabs the active output and hands the whole grab straight to the Quick Access Overlay — no selection step.
  - **"Active output" means the compositor-focused output**, resolved once via `monitor::active_output_name()`, which shells out to `mmsg get focusing-client` and reads `.monitor`. Both `full` and `region` target it. `all-monitors`' `active` flag was rejected: it appears to mean "powered on", true for every docked display, so it can't single one out.
  - **The first-`wl_output`-wins assumption is resolved.** `capture.rs` now binds *every* output, learns each connector name from the v4 `name` event (routing per-output events by the registry id carried as user-data), and grabs the one matching the target — falling back to the first advertised output when the name is `None` or unmatched, so single-display never hinges on the match. The region overlay does the same for its layer surface (previously `get_layer_surface(None, …)`, which could land the frozen backdrop on a different monitor than the grab came from once docked). Passing the same target name to both keeps backdrop and selection surface on one output.
  - **Output selection is compositor-coupled; the pixel grab stays pure Wayland.** On any mmsg failure (absent, no focused client, parse error) `active_output_name()` returns `None` and capture degrades to first-output — for `region` and `full`, mmsg is an enhancement, not a hard dependency. *(Window capture, added 2026-07-20, is the exception: it needs the compositor's window list and has no fallback. See its entry above.)* Pure-Wayland output discovery was considered and rejected: it would mean mapping a layer surface and reading `wl_surface.enter`, a reorder plus a possible one-frame flash in the shipped region path.
  - **The QAO needed one change:** a full 2880×1920 grab overruns wgpu's 2048 max texture side and panicked egui. The *texture* is now downscaled to the preview's physical size; the full-resolution grab is still what saves and copies. Region grabs are small and unaffected. (Found by the single-display smoke — the region path never exercised a full-size texture.)
  - **Verified single-display 2026-07-17** (probe, then reverted): mmsg returns `eDP-1`; the v4 `name` event is delivered and matched — confirmed directly, not merely inferred from the fallback warning staying silent; `frame full` and `frame region` both work end to end. **Unverified until a second display exists:** that mmsg tracks focus across two monitors, and that selection picks the focused output rather than the first-advertised. The Framework would not connect to a monitor on 2026-07-17; deferred, not skipped.
  - **"All outputs" stitched into one image stays deferred.** One display here, so "active" and "all" coincide; multi-output compositing is speculation against absent hardware. Revisit when docked.
- Scrolling capture — **shipped 2026-07-21, verified in use.** `frame scroll` selects a region, captures it at ~30 fps while the user scrolls the real window underneath, stitches the frames as they arrive, and hands one tall image to the Quick Access Overlay. The spike (below) proved the approach on 2026-07-15; this is that design with both stubs replaced.
  - **One key does both, via the socket.** `SUPER+CTRL,Print` → `frame scroll`. If `sock::send("stop")` reaches a listening session, that invocation's entire purpose was to deliver it and it exits; failing to connect is not an error but the answer to a question — no session is running, so become one. This is the roadmap's locked `frame.sock` command bus, now real, carrying exactly one command because only one has a caller.
  - **`Enter`/`Enter` was the spike's *stub*, not the target**, and is worth stating because it reads like a design otherwise. The spike had no keybinding and no socket, so it blocked on stdin — the same category of throwaway as shelling out to `slurp` for the region. Both were listed together for replacement in `SPIKE-FINDINGS.md`. A keyboard trigger is also actively worse here: listening for `Enter` during capture means grabbing the keyboard from a surface sitting over the window being scrolled, which would cost `Space`, `PageDown` and `j`/`k` in that window for the whole capture — ruling out keyboard scrolling of exactly the long documents this exists for. The indicator sets `KeyboardInteractivity::None` for that reason.
  - **Stitching is incremental, which is the one deliberate departure from the spike.** The spike kept every frame and stitched at the end: ~9 MB per frame, about **5.8 GB** across its 639-frame run. Its own timing shows a fold costs ~2.5 ms against a 33 ms frame budget, so folding on arrival is affordable — **~43 MB** for the same run, and the image is finished the instant you stop rather than 1.6 s later. The price is that a capture can no longer be re-stitched with different matcher parameters, which is what the spike's throwaway `restitch` was for.
  - **Every tuned constant carried over unchanged** — `STRIP_MAX` 200, `COL_STEP` 4, `MAX_SEARCH_OFFSET` 300, `BAND_ROWS` 500, `SAD_REJECT_THRESHOLD` 15.0. They took a naive 9454 ms stitch to 1589 ms and are not to be re-derived.
  - **`MAX_ACC_ROWS = 20000`** stops a capture that was started and forgotten, roughly twelve viewports, without limiting a realistic scroll.
  - **The region is re-captured, not cropped from a grab**, via `capture_output_region` — so the loop copies only the selected rectangle. That means converting the selection from the **physical** pixels `overlay::select_region` returns into the **logical** coordinates the protocol takes. At scale 2 those differ by a factor of two and confusing them captures the right shape in the wrong place, which presents as a capture bug rather than a units bug; it has its own test.
  - **`capture_output` was deliberately left alone.** A `Session` was added beside it rather than rewriting it: that function sits behind three live keybindings, and refactoring it for tidiness would have risked the author's daily driver. The overlap is ~40 lines of setup sequencing; the dispatch impls and `buffer_to_rgba`, where the difficulty actually lives, were already shared.
  - **The region indicator was missing and that broke the feature's *feel*, not its function.** Without it a running capture looked identical to nothing having happened, so the natural response was to press the key again — which stopped it. The toggle was working correctly and appeared not to be. Added during verification as four opaque `wlr-layer-shell` bars sitting strictly **outside** the selection: `zwlr_screencopy` copies the composited output, so anything drawn *over* the region lands in every frame. A single surface with a transparent middle should composite away to nothing, but "should" carries a premultiplied-alpha assumption whose failure mode is a silent tint on every capture — four opaque bars cannot contaminate it at all. Empty input region, so clicks and scrolls pass through.
  - **One real bug, found only by using it:** the socket server outlived the capture loop. `qao::show` blocks for the overlay's whole lifetime, so a bus still bound during it swallowed the next keypress — the new process connected, `send` succeeded against a server that would never read it, and it exited having started nothing. The scroll key was dead until the overlay closed. Fixed by dropping the bus before the overlay opens.
  - **The reject path still has no real-data evidence.** `SAD_REJECT_THRESHOLD = 15.0` was never approached during the spike, and a unit test remains the only proof the graceful-gap behaviour works. Unchanged by shipping.
- Self-timer (configurable delay) — **not started. Scoped 2026-07-21; the code is small and the design is where the cost sits.** Less code than blur, the smallest milestone so far; more open questions than blur, which had two. Three things need deciding before any of it is written:
  - **The delay sits in a different place per mode.** For `full` it is obvious: wait, then grab. For `region` and `window` it is not — you almost certainly want to *select first, then* wait, so a menu can be opened or something hovered during the countdown, which is the entire point of a self-timer on a screenshot tool. So this is not one delay inserted once; it is a different insertion point per mode.
  - **How the delay is specified** — a flag (`frame full --delay 5`), a separate subcommand, or a fixed constant with its own keybinding. Note that Anti-patterns forbids inventing CLI flags, but this line has said "configurable delay" since the first planning session, so a flag is sanctioned here rather than a rule violation. The choice also decides whether a fifth `Print` binding is needed or an existing one absorbs it.
  - **Whether the wait is visible.** A silent five seconds is indistinguishable from nothing having happened — precisely the failure that made scroll capture feel broken on 2026-07-21 before the region indicator existed. That mistake is cheap to avoid a second time: `indicator.rs` already puts arbitrary `wlr-layer-shell` surfaces on screen, and `editor.rs` rasterises glyphs through skrifa. A countdown is those two existing pieces together, not new machinery.
  - No default delay has been chosen. Conventional values are 3 or 5 seconds.

### Output
- Save to PNG (configurable output path)
- Copy to clipboard
- Sidecar JSON written alongside the PNG when annotations exist

### Post-capture flow
- Quick Access Overlay: floating preview with save / copy / annotate / discard actions — **shipped 2026-07-16** (`09d9db3`). `frame region` is now interactive rather than a headless one-shot: grab → select → crop → floating `eframe` preview. The command writes nothing on its own; the buttons own it. Save and Copy are **non-terminal** and stay enabled after they succeed — the tick reports what happened, but re-copying is the only recovery if another client takes the clipboard, and a second save writes a new timestamped file. Annotate opens the annotation editor by swapping the same window into an edit mode (milestones 1–4 — see below), no longer a disabled button. Save reuses the timestamped `$XDG_PICTURES_DIR` path; a file picker would mean `rfd`, which pulls a portal or GTK.
  - **Floating depends on a compositor rule that is not in this repo.** `windowrule=isfloating:1,appid:^frame$` in `~/.config/mango/config.conf`. A fresh clone tiles instead of floating, and the scroller then *silently overrides* the overlay's requested size — `with_inner_size` is advisory while tiled. `appid` is a regex and the anchors are load-bearing: unanchored, it would match any future app id merely containing "frame". Confirmed against mangowc 0.14.4 — the **config key** is `isfloating`, while the **`mmsg` JSON field** is `is_floating`; both exist in the binary, so guessing between them yields a rule that parses as nothing.
- Previous-capture quick recall (hotkey or command) — still open; needs a capture history that does not exist
- Last-N capture thumbnails in the overlay — still open; same missing history

### Annotation editor
- **Milestone 1 — two-point vector tools — shipped 2026-07-20, verified in use.** The editor lives inside the Quick Access Overlay: Annotate swaps the same `eframe` window into an edit mode (`qao.rs`, `Mode::{Preview, Edit}`) rather than opening a second window, reusing the loaded image and the `output` save/copy paths. Tools: arrow, line, rectangle (fill toggle), ellipse (fill toggle). A two-row toolbar carries tool select, fill, undo/redo, a fixed six-colour Rosé Pine palette, and a numeric width field (`DragValue`, 1–100 px) — no free colour picker, no config.
  - **The model is the locked one:** an ordered op list with a pointer (`editor.rs`). `push` truncates any redo tail; undo/redo move the pointer; each `Op` is self-contained (type + two points + colour + width + fill). `render` walks the visible prefix from the base image with tiny-skia into a fresh `RgbaImage` — on every *edit*, gated by a `dirty` flag so it is not re-rasterised at frame rate. No diffing, no caching.
  - **tiny-skia, not the egui painter, produces every committed pixel** — reusing the path patterns already in `overlay.rs`, no new crate. The egui painter draws the *in-progress drag only*, as preview feedback; on release the op commits to the list and the flattened render replaces it. Keeping commits on the tiny-skia path in list order is what will let a future blur op sit correctly above one annotation and below another; a painter-on-top overlay could not express that ordering. (This is the reason the locked model rules the painter approach out, recorded now so it is not re-litigated when blur lands.)
  - **The output stays opaque, so premultiplied and straight RGBA coincide on the way back out** — the same fact `base_pixmap` relies on, extended to the flattened result, so the pixmap copies straight into an `RgbaImage`. Holds because blending anything over an opaque screenshot leaves every pixel opaque.
  - **Cursor↔image mapping is the rect↔image ratio, independent of scale:** the texture fills the displayed rect whatever its own resolution, so screen points map to base pixels by that ratio alone. A full grab's editor texture is downscaled to the preview size exactly as the preview texture is (the wgpu 2048 max-side limit), while save/copy flatten at full resolution.
  - **Save/Copy in the editor act on the flattened render**, with the same non-terminal semantics as Preview; Escape/Discard closes. No round-trip back to Preview this milestone.
  - **Deferred past milestone 1:** freehand pencil (with auto-smoothing) and highlighter — both since landed, in milestone 2 below. Still deferred: text — held back because it needs a glyph-rasterisation crate (`cosmic-text`/`ab_glyph`) that nothing else here requires; the destructive ops crop/blur/pixelate (blur and pixelate via `image::imageops`, already a dep); and the sidecar-JSON round-trip, which serialises this same op list.
  - **Verified in use 2026-07-20:** all four tools, live preview, undo/redo (buttons and `Ctrl+Z` / `Ctrl+Shift+Z`), and a saved PNG carrying the annotations, on the compositor. Unit tests cover the op-list semantics and render ordering. Two refinements came out of that pass: the **arrow shaft now stops at the head's base** so its round cap no longer protrudes past the tip (the first cut ran the shaft to the tip, and `LineCap::Round` added a half-disc beyond it), and the width control became the numeric field above (it started as S/M/L presets). Still on the same open footing as `full`/`window`: dual-display is untested.
- **Milestone 2 — freehand pencil and highlighter — shipped 2026-07-21, verified in use.** Two tools join the same toolbar and the same op list: `Pencil`, a smoothed freehand stroke, and `Mark`, a translucent rectangle over a dragged region.
  - **The highlighter is a rectangle drag, not a freehand swipe.** Decided before implementation. `satty`'s highlight tool is a rectangle, so this holds cutover parity; a box is also far more precise over a line of text than a swipe drawn by hand. The cost is that the milestone's two tools share no geometry — the pencil's machinery does nothing for the highlighter — which was accepted rather than overlooked. A freehand highlighter remains possible later; nothing here forecloses it.
  - **Its opacity is a fixed `HIGHLIGHT_ALPHA` (`0x66`), not a toolbar control**, for the same reason the palette is six fixed colours. The op stores the colour that was picked and `render` substitutes the alpha, so the document never holds a derived value — which matters for the sidecar JSON later.
  - **`Op` lost `Copy`.** Forced, not chosen: `Freehand` carries a `Vec<Point>`, so the enum cannot be a bitwise copy. `Clone` covers every use. The only fallout was `draw_op` binding the point list with `ref` while the scalars still bind by value.
  - **A drag is now a point sequence, and the tools differ in how much of it they consume.** `Op::from_drag` takes the sampled path as `&[Point]` and returns `Option<Op>`: the pencil takes all of it, the two-point tools take its ends, and an empty path yields `None` rather than an op describing no geometry. `EditorState.drag` became that same sequence, replacing the single anchor — one field with one lifecycle, rather than an anchor plus a parallel stroke buffer that could go stale.
  - **Decimation is baked into the op; smoothing is not.** A pointer sampled at 120 Hz oversamples badly enough that consecutive frames land on the same pixel, so `decimate` drops samples within 2 px of the last survivor, always keeping both endpoints. That is noise removal — nothing later wants the duplicates back. The *curve* is `render`'s business: `smooth_path` walks quadratic Béziers through segment midpoints at draw time, so the stored op keeps the path as drawn. Baking the curve in would freeze a rendering choice into the document, and the sidecar-JSON round-trip will want the samples.
  - **One path per stroke, for smoothing and joins — not for alpha.** The first framing of this decision argued it from translucency: that N separate segments would double-composite and bead at every overlap. That argument does not apply here and was corrected during planning. The highlighter is a rectangle with no self-overlap, and the pencil is opaque, so double-compositing would be invisible either way. The single path is still required, because the Béziers have to span segment boundaries to smooth anything and a single path lets the stroker resolve each join once. The translucency argument only becomes live if a freehand highlighter is ever added.
  - **The opacity invariant survived, and is now tested rather than asserted.** `render` copies tiny-skia's premultiplied buffer straight into a straight-RGBA image, which is only valid because every pixel is opaque. A translucent op is the first thing that could have broken that; blending translucent over opaque still yields opaque, and a test now pins it instead of a comment claiming it.
  - **A click with no drag leaves a dot.** `smooth_path` emits a zero-length segment for a one-point stroke and relies on the round cap to rasterise it. Whether tiny-skia does that was genuinely unknown at planning time — it was written as a test rather than an assumption, and the test passes.
  - **`MIN_WINDOW_W` rose 340 → 520** so six tools plus the fill checkbox and undo/redo fit the row without wrapping.
  - **Two accepted consequences.** The pencil's live preview is the raw polyline while the commit is the smoothed curve, so a stroke settles slightly on release — not reported as visible in use. And two overlapping highlights composite twice, so the overlap is darker; that is what an ordered op list means, and hiding it would require the highlighter to merge with its neighbours, which no other op does.
  - **Verified in use 2026-07-21:** both tools end to end on the compositor, against a checklist covering smoothing on fast corners, the click-with-no-drag dot, tint readability over text, overlapping highlights, undo/redo across mixed op types, and a saved PNG at full capture resolution. Unit tests cover decimation, freehand rendering including both degenerate lengths, the tint, and the opacity invariant. Dual-display remains untested here as everywhere else.
- **Milestone 3 — blur — shipped 2026-07-21, verified in use.** A `Blur` tool drags a rectangle and makes everything already drawn inside it unreadable. The first destructive op.
  - **The in-order render path is no longer a promise.** Since milestone 1 the argument for routing every committed pixel through tiny-skia in list order has been that a future blur would need to sit correctly above one annotation and below another. Blur is that op, and it needed *no new machinery* to do it: `render` already walks the visible prefix, so by the time `draw_blur` runs, the pixmap holds every earlier op composited onto the base. It reads the buffer it is handed. A test now pins both directions — an op before a blur is blurred, the same op after it stays sharp — where previously only the reasoning existed. **The design claim held.**
  - **Blur only. Pixelate was dropped, deliberately.** Both make a region unreadable and the author would reach for whichever is bound; carrying two tools for one job failed the BYOS test. It remains trivially addable — the same `Op` shape with two `Nearest` resizes in place of the filter — if hard blocks ever turn out to say "deliberately redacted" in a way a blur does not.
  - **Strength is a fixed `BLUR_SIGMA = 12.0`, not a control.** Redaction is binary: the text is unreadable or the op failed, so there is little to tune. Absolute in base-image pixels rather than scaled to the region, since region, window and fullscreen captures are all cropped from the same physical grab and a pixel means the same thing in each. 12.0 was a guess at planning time and is the one number no unit test could settle; **confirmed unreadable against real small text on the compositor, 2026-07-21.**
  - **`Op::Blur` carries no colour, width or fill** — a destructive op adds no ink, it names a region and a position in the list. The consequence is that the palette and width field stay visible but inert while `Blur` is selected. Greying out the whole second row per-tool is a real UI change and was deliberately *not* made as a side effect of this milestone; `Fill` already sets the precedent if it is wanted later.
  - **The mechanics:** `clamped_int_rect` rounds the drag corners outward and clips them to the canvas, so a blur never covers less than it was dragged over and a partly-offscreen drag still works; `Pixmap::clone_rect` lifts the region out; the buffer moves into an `RgbaImage` untouched, since both are tightly packed RGBA8; `imageops::fast_blur` runs; `draw_pixmap` puts it back at the same origin. `fast_blur` documents that it expects premultiplied alpha, which is exactly what a `Pixmap` stores — the two agree with no conversion, which is worth knowing because it otherwise reads like a latent bug. Every step is fallible and every failure is a silent return: a degenerate or wholly-offscreen drag has nothing to blur.
  - **The drag preview is a neutral hairline marquee, not the palette colour and not a live blur.** Colour has no meaning for this op, so previewing the selected one would lie. Running the filter every frame would have been the first thing in this editor to cost real time, and the preview convention was already responsiveness over fidelity.
  - **The tool row now wraps** (`horizontal_wrapped`) rather than overflowing, with `MIN_WINDOW_W` 520 → 580 for seven tools.
  - **Verified in use 2026-07-21:** small text rendered unreadable at the saved full resolution — not merely in the downscaled preview, which is the check that actually matters for redaction — ordering visible both ways against a drawn arrow, blur over an existing highlight, clean undo with no residue, a partly-offscreen drag, and the toolbar fitting.
- **Milestone 4 — text — shipped 2026-07-21, verified in use.** A `Text` tool typed directly on the canvas: click to place a caret, type, `Enter` commits, `Escape` cancels. The last tool of the original set. **This closes `satty` tool parity.**
  - **Font: Atkinson Hyperlegible Regular, embedded as a 29 KB OTF** (`assets/`, with its OFL text). Built by the Braille Institute with Applied Design Works for low-vision readability. It is both the smallest sans-serif on this machine — measured against a dozen installed candidates, next nearest 100 KB, Roboto's variable face 2.4 MB — and the one designed for the failure mode annotation text actually has: a label over arbitrary screenshot pixels, where `O`/`0` and `I`/`l`/`1` confusion is the real risk.
  - **The free alternatives were rejected on weight, not size.** `eframe`'s `default_fonts` already embeds Ubuntu-Light and Hack-Regular at zero marginal bytes. Ubuntu is a *Light* weight — thin strokes are the worst case over a busy capture — and Hack is monospace. "Already free" was a strong argument when the alternative cost 361 KB; against 29 KB it stopped being one.
  - **The CFF assumption was checked before anything depended on it, and held.** `OutlinePen`'s five methods map 1:1 onto `PathBuilder` (`curve_to` → `cubic_to`), and `OutlineGlyph::draw` dispatches CFF and glyf through the same pen — so the 29 KB OTF works and the 53 KB TTF fallback was never needed. Recorded because the Locked Dependencies preamble asks for exactly this: confirmed against the crate source, not against a hopeful `Ok`.
  - **The binary grew 250 KB, not the ~29 KB predicted — the prediction was wrong.** The font is 29 KB of it; the rest is skrifa's outline machinery (CFF and glyf parsing, metrics, autohinting) newly linked because `epaint` never reached that part of the API and the linker had been stripping it. "No new crates compile" was true; inferring "no meaningful new bytes" from it was not. 1.3% of a 19 MB binary, so not a problem — but the reasoning was faulty and the number is measured, not estimated. *(The attribution to skrifa's outline code is inferred from what changed, not from a symbol-level measurement.)*
  - **`pos` is the top-left of the line box, not the baseline**, with the baseline derived at render as `pos.y + ascent`. **How a click maps onto `pos` was wrong at first, and the fix is a correction, not a tweak.** The original reasoning — "a click should put text below-right of the cursor, the way a caret implies" — reasoned about a *text-editor caret*, whose bar starts at the line-box top. The relevant object is the *I-beam pointer*, which points with its **middle**. Anchoring the line-box top at the click therefore placed every label about half a line low. The click is now lifted by `text_center_offset`, half of `ascent + |descent|`, so the text straddles the cursor and the caret centres on it. The op's data model never changed; only the click→anchor mapping did, which is where the mistake actually lived.
  - **The bug was invisible until the cursor was fixed.** With an arrow pointer there was no reference point to be wrong against, so the offset read as normal. Changing the pointer to an I-beam on 2026-07-21 exposed it immediately. Worth remembering as a class: a cursor that lies about a tool also hides that tool's placement errors.
  - **The pointer becomes an I-beam over the canvas while `Text` is selected**, scoped to hover so the toolbar keeps a normal arrow. The other six tools stay on the default arrow — they drag, and an arrow is honest about that. A crosshair for them was considered and not adopted; the region overlay's `+` marker is a separate surface with its own reasons.
  - **The pen adapter's whole job is the Y flip.** Font outlines are Y-up from a baseline origin, tiny-skia is Y-down from the top-left, so every emitted point becomes `pen_x + x`, `baseline - y`. Flipped the wrong way, glyphs render *above* the anchor and still look like perfectly good text — which is why it has a positional test rather than a visual check. It was right first time.
  - **Glyphs are filled paths, not a rasterised mask** — the same `fill` every other op uses. Text therefore composites identically to everything else, and a later blur smears it like anything else.
  - **Text is the first tool a drag does not produce.** `from_drag` returns `None` for it, so dragging with `Text` selects nothing and commits nothing; the op is built by `commit_text` instead.
  - **Typing is a modal state, and it takes two shortcuts away from their usual jobs.** While an entry is open, `Escape` cancels it instead of closing the overlay — otherwise a mistyped label would discard the whole annotation — and `Ctrl+Z` is suppressed so it cannot reach back and undo a previous op mid-word. Both were global reads with no notion of focus before this milestone. Switching tools commits an open entry rather than dropping it; an empty buffer commits nothing, since an empty op would still consume an undo step.
  - **The font is registered with egui under its own family name** so the canvas preview is set in the face the commit will use, without disturbing the toolbar's font. egui still does its own layout while the commit uses skrifa advances, so spacing differs slightly — the same class of accepted mismatch as the pencil settling onto its smoothed curve.
  - **Size is a numeric field, superseding this roadmap's earlier "predefined style set".** That wording predated milestone 1, where the author replaced S/M/L stroke-width presets with a number; the same preference applies here and the old line was wrong to leave standing.
  - **No legibility outline, chosen deliberately — with a known cost.** Light palette entries (text, gold, foam) are hard to read over light screenshot regions, which is much of a typical UI capture. The mitigation is picking `base` from the palette. If that grows annoying the fix is stroking the glyph path before filling it, three lines, since both helpers exist.
  - **Verified in use 2026-07-21.** Unit tests cover orientation, advance accumulation including the space-has-no-contours case, the empty string, a character with no glyph in the font, and the opacity invariant.
- **Destructive ops:** blur shipped in milestone 3 above. Pixelate dropped as redundant. Crop remains, tier undecided.
  - **Crop's tier is deliberately undecided as of 2026-07-21.** It is the most invasive work remaining — canvas dimensions change mid-list, coordinates fork before and after the op, and the texture, window and cursor mapping all key off `doc.base` today. Against that, `frame region` already makes the author draw the capture rectangle before the editor opens, so crop is a second chance at a decision just made deliberately; `satty` needed it only because `grim -g "$(slurp)"` handed it a fixed region. Rather than predict, the call waits until blur, pixelate and text have seen real use and it is clear whether crop is ever missed. **Until then Tier 1 is not fully bounded, and that is intentional.**
    - **First signal, 2026-07-21: not missed.** Reported after the first afternoon of real use following the cutover. That is hours, not weeks, and it is recorded as a signal rather than a verdict — the honest reading is "nothing has argued *for* crop yet", not "crop is unnecessary". Leaning toward Tier 2; revisit after a longer stretch, or the first time a capture is framed wrongly and there is no way to fix it.
  - **Crop cannot reuse `overlay.rs`.** Asked and answered 2026-07-21. `overlay.rs` is a standalone layer-shell application — its own Wayland event loop, `run(…) -> Option<Rect>`, returning one rectangle and exiting before the editor exists. Driving it from inside the running `eframe` app would mean bridging layer-shell and xdg-shell, which the locked GUI split forbids, and would raise a fullscreen overlay to select a region of an image that is not on screen at those coordinates. The reuse worth having is already in the editor: `Tool::Rect` and `Tool::Highlight` drag two corners over the canvas, and crop is one more arm in `from_drag`. Selection was never the expensive part.

## Pre-v1 Spike: Scrolling Capture Feasibility

> **Delivered 2026-07-21.** This section is kept as the record of *why* the approach was chosen; the shipped feature is described under Capture above. Read this for the reasoning, that for the behaviour.
>
> **Status: PASSED (GO) — 2026-07-15.** The capture-and-stitch loop produces readable, seam-free output on text-heavy content; all four success criteria met. SAD proved adequate — the `rustfft` fallback is **not** needed. Stitch clears the 2 s target (1589 ms on a ~5-viewport stress run). One noted weakness: periodic/low-texture content (e.g. blocks of identical rules) is ambiguous for SAD and can mis-stitch without a reject — degenerate content, deferred. Full write-up in `SPIKE-FINDINGS.md`. The throwaway spike code lives in `a6762c1` and was retired from the tree by `bf0c4e4` — it is *not* in `src/main.rs`.

Wayland has no compositor primitive for capturing offscreen surface content. The approach is select-area-then-user-scrolls, with frame stitching via the `wlr-screencopy` loop.

**Goal:** prove the capture-and-stitch loop produces a usable, seam-free image on representative content before committing v1 to a schedule that depends on it.

**Approach:**
- User selects region, presses bound hotkey to start, scrolls naturally, presses hotkey again to stop (via Unix socket signal to the running instance).
- Frames captured at ~30 fps via `wlr-screencopy`.
- Frame-to-frame overlap detected by sum-of-absolute-differences (SAD) match: take a horizontal strip from the bottom ~20% of frame N, slide it vertically over frame N+1, accept the offset minimising SAD.
- Reject pairs where best-match SAD exceeds a confidence threshold — drop the frame, accept a potential gap.
- Stitch: append the non-overlapping bottom portion of frame N+1 to the accumulated image.

**Fallback if SAD proves unreliable:** phase correlation via `rustfft`. More robust to noise; FFT-based image registration is well-understood and pure-Rust.

**Success criteria:**
- Readable stitched output for text-heavy content (docs, code, web pages) over 2–3 viewport scrolls.
- No visible seams in typical cases.
- "Done" hotkey to image-ready in under 2 seconds for the typical case.
- Failure modes degrade output gracefully (seams, gaps), not crashes.

**Out of scope even if the spike succeeds:** animated content, infinite-scroll feeds where content loads mid-capture, backward scroll, any content where the viewport changes between frames for reasons other than scroll position.

## Locked Dependencies

Settled during the initial planning session. Versions to confirm against latest at implementation time.

**Behavioural claims in this section were written from planning-time assumption, not from running the crates.** The crate choices are locked; what the crates *do* is not established until an implementation has exercised it. Verify against the crate source and against observable end state — not against a returned `Ok`. The clipboard correction below is what one unverified sentence cost when a later session read it as fact.

**Wayland capture:**
- `wayland-client` + `wayland-protocols-wlr` (raw), targeting `zwlr_screencopy_v1`. Pure-Rust backend, no `libwayland-client.so` runtime dependency.
- `libwayshot` — **fallback retired 2026-07-16.** It was contingent on the spike revealing the raw approach was miscalibrated. The spike passed (GO), and region capture then shipped on the raw stack, so the condition can no longer fire. Recorded as a closed decision, not a live option.

**GUI framework:**
- `eframe` (egui + winit + wgpu) for xdg-shell surfaces: annotation editor, Quick Access Overlay.
- `skrifa` for glyph outlines, plus an embedded Atkinson Hyperlegible OTF — **shipped 2026-07-21** (milestone 4). Promoted from a transitive dependency of `epaint`, so it compiles nothing new, but it is pinned to a version `eframe` chose and an `eframe` upgrade can desync it. Reasoning, measurements, and the 250 KB binary surprise live in the text entry under Annotation editor; do not restate them here.
- Raw `wayland-client` + `tiny-skia` for `wlr-layer-shell` surfaces: region-selection overlay, and any Tier 2 layer-shell work (freeze-screen mode, floating screenshots).
- **The two stacks run sequentially in one process — verified 2026-07-16 (`09d9db3`).** winit's event loop starts cleanly *after* the raw layer-shell overlay has torn down; `frame region` does exactly this on every capture. This was the main risk in the QAO plan, since nothing established that the second could follow the first on one connection. Running the overlay as a re-exec'd process was the fallback: it is not needed, and is a closed option rather than a live one. This is sequencing, not bridging — `winit` still has no layer-shell support, so the surface split above stands.
- **Feature selection is deliberate.** `eframe`'s defaults pull `x11` — against the Wayland-first principle — plus `web_screen_reader` (dead weight in a native binary) and `accesskit` (not asked for). Only `default_fonts`, `wayland`, and `wgpu` are enabled. It builds clean without the rest. `persistence` is **not** a default, so `with_app_id` (which would otherwise select the persistence-file location) causes no state files.
- `eframe` does bundle wgpu, and wgpu wants Vulkan. This is precisely why the principle above is worded as "requires no system library Void does not already provide" rather than as static linking: Mesa/Vulkan is already required by MangoWM via wlroots, so the overlay adds no dependency the compositor does not already impose. Recorded as an accepted cost, not a discovery to be made again at link time.
- **Linkage measured 2026-07-22.** The binary's only link-time (`NEEDED`) dependencies are `libgcc_s.so.1`, `libm.so.6`, `libc.so.6`, and `ld-linux-x86-64.so.2`. Everything else is `dlopen`ed by name at runtime: `libvulkan.so.1` (entered through `vkGetInstanceProcAddr`), `libEGL.so.1`, `libwayland-client.so.0`, `libwayland-egl.so.1`, and `libxkbcommon.so.0`. That is why `ldd` output is four lines long, and why "static binary" was always the wrong word for this — the wording was corrected on the same date. Verify with `objdump -p target/release/frame | grep NEEDED`; note that grepping the binary's strings for library names needs no `\b` anchor, since Rust concatenates string literals and the name will sit mid-run.
- **The weight of that bundle is now measured, and the exit is written down.** 182 of 198 crates and most of a 19.7 MiB binary (2026-07-22). That is not grounds to unlock anything on its own; Tier 2 records the hand-written `wayland-client` + `tiny-skia` alternative and, more importantly, what evidence would justify it. Read that entry before proposing a swap, so the lock and the candidate are not mistaken for a contradiction.

**Clipboard:**

> **Correction — 2026-07-16.** This section previously asserted that `wl-clipboard-rs` "forks a background helper to serve paste requests after the process exits when required." It does not, and never did. The claim was recorded at planning time without running the crate, and was then read as settled fact during task 6: `src/output.rs` was written to it, and shipped a `frame region` that printed `copied to clipboard` while leaving the clipboard empty. It cost a debugging round, and the false confirmations are worth remembering — `copy()` returned `Ok`, the PNG was on disk, and neither was evidence of anything. The only real test was whether the offer outlived the process. Corrected behaviour below.

- `wl-clipboard-rs`. Pure-Rust backend (do not enable the `native_lib` feature). Handles both interactive-overlay and headless-CLI copy contexts through the same code path.
- **A Wayland clipboard offer is served live by the client that makes it**, so it dies with that process. `wl-clipboard-rs` serves it from a thread inside the calling process (see `Options::foreground` in the crate's `copy.rs`) — there is no forked helper. An `Ok` from `copy()` means the offer was registered, not that it will outlive you.
- **`frame` re-execs itself** as a detached `__serve-clipboard` child, which reads the PNG from stdin, claims the offer, and serves until another client takes the clipboard. The child reports success over a pipe before it starts serving, so a genuine failure still reaches the parent's exit code.
- **Re-exec rather than `fork()`:** the Quick Access Overlay copies from an `eframe` (winit/wgpu) process, and `fork()` carries only the calling thread into the child — a mutex held by any other thread at that instant, the allocator's included, stays locked forever in a child that then allocates in its serve loop. `exec` resets the address space, so one code path stays safe from both the single-threaded CLI and the multithreaded overlay.
- **Verified 2026-07-16:** `frame region` → drag → release leaves a PNG on disk and a byte-identical `image/png` on the clipboard, served by a child reparented to `PPID 1` that outlives the capture.
- **The re-exec reasoning has now been exercised — 2026-07-16 (`09d9db3`).** The argument above was written from reasoning alone, before any overlay existed to test it. `copy()` called from inside the multithreaded `eframe`/wgpu process leaves an `image/png` byte-identical to the file on disk, read back *after* the process exited, served by a child at `PPID 1`. The fork-versus-exec argument holds; it is no longer a prediction.

> **Do not verify the clipboard by asking for a paste-back.** Copying the terminal output in order to report it makes the *terminal* take the selection, which evicts our offer and makes a working clipboard look broken. This produced a convincing false alarm on 2026-07-16: `wl-paste --list-types` showed no `image/png` and no helper was alive, and both were correct behaviour — `serve()` is documented to exit when another client takes the clipboard. The tell was that the clipboard held the exact output text, owned by `application/glfw+clipboard-<pid>` (GLFW is kitty's toolkit). **Method that works:** run the command so its output reaches the reader without a copy, then read `wl-paste` from a separate shell while the offer is live, and compare bytes against the file. Related: `pgrep -f` matches the shell running the check — use `pgrep -x frame`, or a phantom process will confirm whatever you already suspect.
